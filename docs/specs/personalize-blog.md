# 第一阶段：个人信息替换

## 背景

当前项目仍保留开源模板的默认个人信息，例如 `Kai Space`、`Demo Author`、示例邮箱、示例 GitHub、About 页模板文案和部分首页硬编码品牌文案。第一阶段目标是先把这些公开展示信息替换成用户自己的个人博客信息。

## 目标

- 使用 `docs/personal-profile-template.md` 作为个人信息来源。
- 替换站点基础信息、About 页、文章默认作者、首页/文章归档/项目页的明显模板文案。
- 保持现有视觉风格、目录结构和功能边界。

## 范围

会改：

- 站点元信息和导航数据。
- About 页展示文案。
- 新文章模板里的默认作者信息。
- 首页、文章归档、项目页中明显属于模板的品牌或个人文案。
- 必要时调整文章作者卡片中的社交链接占位。

不会改：

- 登录认证逻辑。
- Astro BFF 和 `/api/*` 代理。
- Node helper 后端。
- Beszel 数据库接入。
- 目录结构。
- RSS/sitemap 功能；当前项目未实现，后续单独规划。

## 方案

1. 等用户填写 `docs/personal-profile-template.md`。
2. 只按已填写字段替换；空字段不猜测。
3. 优先改集中配置，再改页面中的硬编码文案。
4. 每次改动保持小范围，避免同时重做 UI 和数据结构。

## 安全与隐私

- 不在个人信息文档中填写密码、token、私钥或敏感凭据。
- 不提交 `.env`、本地快照、Beszel DB 或 CLI 认证文件。
- 不暴露 helper backend 端口。

## 验收标准

- `pnpm build` 通过。
- 首页、关于页、文章列表、文章详情、标签页、项目页能正常打开。
- 移动端没有明显横向溢出。
- 深色模式下文字和按钮可读。
- 不再明显显示模板个人信息：`Demo Author`、`hello@example.com`、`github.com/example`。
- 如果仍保留 `Kai Space` 作为项目名，必须是用户明确选择；否则应替换为用户的网站名称。

## 后续事项

- 决定是否隐藏订阅、API、服务器、AI 用量和 IP 风险模块。
- 决定是否新增 RSS、sitemap、SEO metadata。
- 决定是否替换 favicon 和头像图片。
